Critical

Issue 1 — window_tick.py is a God Module for Per-Window Polling

handlers/window_tick.py — 616 lines

Strength: HIGH Distance: LOW Volatility: HIGH

This file orchestrates all per-window polling: dead-window detection, transcript discovery, interactive UI, status updates, multi-pane scanning, and passive shell relay. It imports from 24 distinct modules and contains 6 identical expressions resolving the current window provider — none extracted to a helper.

Private API Access (window_tick.py:335)

from ..tmux_manager import _has_insert_indicator, notify_vim_insert_seen
if _has_insert_indicator(vim_text):
    notify_vim_insert_seen(w.window_id)

_has_insert_indicator is a private function. Refactoring vim-insert detection in tmux_manager will silently break window_tick.

6 Identical Provider Resolution Expressions

# Repeated at lines 152, 205, 226, 325, 448, 557 — no variation:
provider = get_provider_for_window(
    window_id, provider_name=session_manager.get_window_provider(window_id)
)

Each copy is an independent context-loading event for an AI making changes here.

Recommendation: Promote _has_insert_indicator and notify_vim_insert_seen to public API or move vim-insert detection into the polling layer. Extract a _get_provider(window_id) helper to replace all 6 duplicates.

Issue 2 — hook_events.py Has 15 Module Dependencies, Half Hidden

handlers/hook_events.py — 385 lines, 15 distinct module imports (9 deferred)

Strength: HIGH Distance: LOW Volatility: HIGH

Hook event dispatch is the highest-volatility module (new hook types are added as Claude Code evolves). It has the broadest import surface in the codebase. Balance rule: HIGH strength × HIGH volatility → UNBALANCED.

Cross-Subsystem Boundary Violation

# hook_events.py:180 — inside _handle_stop()
from .periodic_tasks import run_broker_cycle
run_broker_cycle(bot, user_id, thread_id)

Event dispatch reaches directly into polling infrastructure. Adding a new hook event requires understanding both the event layer and the polling infrastructure.

Same Module Imported 4 Times (Deferred)

# hook_events.py lines 60, 129, 236, 315 — different function bodies:
from .message_queue import enqueue_status_update

Hidden fan-out: only visible by scanning all function bodies at runtime, not by reading imports.

Recommendation: Move asyncio to top-level. Consolidate message_queue deferred imports to module level. Replace run_broker_cycle direct call with a registered callback to break the hook→polling boundary.

Issue 3 — session_map.py Bypasses WindowStateStore and ThreadRouter APIs

session_map.py, window_state_store.py, thread_router.py

Strength: HIGH (intrusive) Distance: LOW Volatility: MEDIUM 
session_map.py:194  for w in window_store.window_states          # iteration
session_map.py:199  window_store.window_states[w].session_id     # field read
session_map.py:204  del window_store.window_states[wid]          # deletion
session_map.py:294  if window_id in window_store.window_states:  # membership test
session_map.py:295  del window_store.window_states[window_id]    # deletion
session_map.py:188  for user_bindings in thread_router.thread_bindings.values()

WindowStateStore already provides clear_window_session(), prune_stale_window_states(), and get_window_state(). Direct dict access bypasses these and prevents adding side effects (cache invalidation, listener notification) to window removal.

Recommendation: Make window_states private (_window_states). Add remove_window(window_id) and iter_window_ids(). Expose ThreadRouter.iter_bound_windows(). Replace all 6 direct accesses in session_map.py.

Issue 4 — parse_session_map Logic Duplicated Across Two Files

session.py (lines 50–88), session_map.py (lines 34–72)

Strength: HIGH (duplication) Distance: LOW Volatility: MEDIUM

Both files define parse_session_map() and parse_emdash_provider() with identical implementations. A change to emdash's session naming convention must be applied to both or behavior diverges silently. session_map.py is the canonical home.

Recommendation: Delete both functions from session.py. Update session_monitor.py to import from session_map. Zero-risk, one-file change.
Medium

Issue 5 — status_bubble.py Crosses into Polling Layer

handlers/status_bubble.py, handlers/polling_strategies.py

Strength: LOW-MEDIUM Distance: MEDIUM (different subsystems) Volatility: LOW 
# status_bubble.py:189-195
from .polling_strategies import terminal_screen_buffer
keyboard = build_status_keyboard(
    window_id, history=history,
    rc_active=terminal_screen_buffer.is_rc_active(window_id),
)

Low volatility makes this tolerable but architecturally surprising — the display layer should not know how RC detection works. Secondary issue: status_bubble accesses 4 internal fields of ClaudeTaskSnapshot (total_count, done_count, open_count, items), coupling display to the task state schema.

Recommendation: Pass rc_active: bool as a parameter to build_status_keyboard(). Add a format_task_summary() → str method to ClaudeTaskState to encapsulate display logic.

Issue 6 — shell_capture ↔ shell_commands Mutual Circular Dependency

handlers/shell_capture.py, handlers/shell_commands.py

Strength: HIGH (mutual) Distance: LOW Volatility: MEDIUM

shell_context.py was correctly extracted to break the static cycle, but shell_capture._maybe_suggest_fix() still calls shell_commands.show_command_approval() via a runtime deferred import. Additionally, shell_capture.py calls asyncio.create_subprocess_exec("tmux", ...) directly, bypassing tmux_manager.

Recommendation: Introduce a CommandApprovalCallback protocol. shell_capture accepts it as a parameter — eliminates the circular dependency architecturally. Route the capture-pane call through tmux_manager.

What Is Working Well

Data contract module

message_task.py — Pure frozen dataclasses, zero project imports. Any module can import without loading other project code. Textbook contract coupling.

Thin orchestrator

polling_coordinator.py (87 lines) — Delegates all work, no logic leakage. The result of recent refactoring from a 522-line monolith.

Clean data ownership

spawn_request.py — Pure CRUD with no handler/Telegram/config dependencies.

Provider facade

providers/__init__.py — Lazy registration avoids circular imports. Detection, factory, and launch command resolution cleanly separated. Adding a new provider requires changes only within providers/.

Registry pattern

topic_state_registry.py — Self-registration decorator replaces 14+ deferred imports in cleanup.py. Correct architectural direction for cross-cutting concerns.

Recommended Priorities

  1. Promote or relocate private tmux functions — _has_insert_indicator, notify_vim_insert_seen in tmux_manager.py. Clearest encapsulation violation; window_tick.py touches 24 modules.
  2. Extract _get_provider(window_id) helper in window_tick.py — eliminates 6 duplicate 3-line expressions; immediately reduces AI change-context.
  3. Delete parse_session_map duplicate from session.py — zero-risk; reduces emdash change sites from 2 to 1.
  4. Make window_store.window_states private — forces session_map.py to use the existing clear_window_session() and prune_stale_window_states() methods.
  5. Pass rc_active as a parameter to build_status_keyboard() — severs the status_bubble → polling_strategies cross-subsystem import.
